# Vehimap i18n and formatting

Vehimap 2.0 is entering a localization-first phase before larger new features and before Android UI work. The first supported languages are English and Czech. English is the neutral fallback language and Czech remains the primary development and QA language.

## Source of UI text

- User-facing UI strings belong in `Vehimap.Application/Resources/Strings.resx` and `Strings.cs.resx`.
- `Strings.resx` is the English fallback.
- `Strings.cs.resx` contains the Czech translation and is used for `cs` / `cs-CZ`.
- Every new resource key must be added to both files in the same change.
- Resource keys use dotted, stable names such as `Settings.Title`, `VehicleEditor.Name` or `MainMenu.App.Settings`.
- Keep keys semantic. Do not encode the exact English wording into the key.
- Avoid reusing one key for different contexts if grammar or screen-reader wording may diverge later.

## Translator workflow

Translators can edit the `.resx` files directly, but a dedicated editor is strongly preferred. A practical option is [Local ResX Editor](https://apps.microsoft.com/detail/9pkht426fq1k), which can edit `.resx` resources without requiring Visual Studio or manual XML edits.

When handing files to translators:

- Give them both `Strings.resx` and `Strings.cs.resx` so they can compare source and target text.
- Ask them to preserve placeholders such as `{0}`, `{1}` and `{2}` exactly.
- Ask them not to translate resource keys.
- Ask them to keep accelerator hints such as `Ctrl+S` and `Escape` consistent with the actual UI.

## Settings

The first i18n settings are intentionally small and safe:

- language: `system`, `cs-CZ`, `en-US`
- thousands separator: language default, space, comma, dot, none
- decimal separator: language default, comma, dot
- distance unit: kilometers, miles
- fuel volume unit: liters, US gallons, imperial gallons

Changing language is saved immediately, but the full UI uses the new language after restarting Vehimap. This avoids partial live reload states while the application is still in the 2.0 nightly phase.

Language defaults are only used for missing settings. Existing user choices are never overwritten automatically. The current defaults are:

| Language | Thousands separator | Decimal separator | Distance | Fuel volume |
| --- | --- | --- | --- | --- |
| Czech (`cs-CZ`) | none | comma | kilometers | liters |
| English (`en-US`) | comma | dot | miles | US gallons |

Manual language changes in the Settings dialog update only the language choice. They do not reset already selected separators or units.

## Installer language handoff

The Windows Inno Setup installer offers English and Czech. After installation it writes a one-shot `installer-preferences.json` file into the channel data folder, for example `%APPDATA%\Vehimap Nightly`.

Vehimap reads this seed during startup after SQLite migration/load:

- if `app/language` is missing, it is set from the installer language
- missing separators and unit settings are filled from the language defaults above
- existing settings stay unchanged
- the seed is removed after successful handling

The installer must not write `settings.ini`. In Vehimap 2.0, `settings.ini` is a legacy migration input only; active settings are stored in `data/vehimap.db`.

## Invariant storage

Storage stays invariant:

- SQLite stores stable internal values.
- Distances remain metric internally unless a future schema migration explicitly says otherwise.
- Fuel volume remains liters internally unless a future schema migration explicitly says otherwise.
- Number and unit preferences affect display, input parsing and exports, not the meaning of existing database values.
- User-facing distance settings, such as the maintenance reminder distance, must display and parse the selected distance unit but persist the normalized kilometer value.
- Editor fields for history odometer, fuel odometer, maintenance intervals, maintenance completion odometer and last-service odometer display the selected distance unit and save normalized kilometers.
- Fuel editor volume displays the selected volume unit and saves normalized liters.
- Switching units in Settings is not a data migration. Existing values are re-rendered through the new unit preference, and saving without semantic changes must not multiply or otherwise shift the stored meaning.
- Switching number separators is also not a data migration. Visible values are re-rendered through the new separator preference while storage stays invariant.
- The resolved thousands separator and decimal separator must not be the same non-empty symbol. Ambiguous combinations such as comma/comma or dot/dot are rejected before parsing user-entered values.

This rule is important for backup compatibility, Android reuse and future imports from older Vehimap builds.

## Formatting rules

- Use `IAppNumberFormatService` for localized number formatting and parsing.
- Use `IAppUnitFormatService` for distance and fuel volume display or conversion.
- Do not call `ToString()` on user-visible numbers without an explicit culture or formatting service.
- Keep units visible in UI labels, accessible names or help text when ambiguity is possible.

## Accessibility

Localized UI text must still satisfy the accessibility checklist in `ACCESSIBILITY.md`:

- interactive controls need stable `AutomationId`
- accessible names must be human-readable in the active language
- help text should explain examples and keyboard shortcuts when needed
- placeholders are examples, not replacements for accessible names

## Current pilot coverage

The second localization pass covers the most visible shell surfaces:

- Settings, About and Vehicle Editor
- the main menu and application-level menu items
- the vehicle list left panel, filters and data folder labels
- the main workspace tab names and their `Open in window` actions
- selected shell status, About diagnostics labels and About clipboard/status feedback

The next passes extend the pilot to the static surfaces of the core evidence editor dialogs and their workspace views:

- dialog help text, field labels, placeholders, button names and accessibility names come from resources
- unit-aware labels such as odometer, fuel volume and maintenance distance interval are formatted through resource keys with the active unit label
- workspace headings, search/sort controls, list names, action buttons, fuel analysis section headings, attachment actions and list item types come from resources
- the Vehicle Detail workspace uses resources for vehicle action buttons, related-record navigation, list names and list item types while dynamic vehicle summaries stay in view-model/domain localization work for later
- Global Search and Timeline workspace static surfaces use resources for filters, search fields, sorting, list names, item types, detail headings and keyboard help; generated result details remain in the later dynamic/domain localization pass
- the maintenance completion dialog uses resources for title, help text, field labels, unit-aware odometer wording, validation errors and save/cancel names
- global safety dialogs such as confirmation prompts and the SQLite data set health check use resources for titles, help text, button names and copy/status feedback
- standalone workspace window chrome uses resources for accessible window names, help text and close buttons; the hosted workspace content remains covered by its own pass
- the accessible `Tray actions` window uses resources for its title, description, section headings, button names, help text and view-model defaults; the native tray menu remains a short convenience menu and is not the screen-reader-first path
- dynamic workflow status messages, item details and domain-generated analysis/audit text remain intentionally out of this pass

These areas are guarded against new Czech hardcoded UI text. Larger workspace views such as Dashboard, Smart Advisor and remaining workflow status messages are intentionally left for later passes so each step stays reviewable.

## Development rules

- Commit messages are always English.
- New UI text in the pilot localization areas must be added via resources, not hardcoded directly in XAML or C#.
- The current pilot areas are Settings, About, Vehicle Editor, the main shell surfaces listed above, History/Fuel/Maintenance/Reminder/Record editor dialogs, the Maintenance Completion dialog, the Vehicle Detail, Global Search and Timeline workspaces, the static History/Fuel/Maintenance/Reminder/Record workspace surfaces, Confirmation, Data Store Health, standalone workspace window chrome and the accessible Tray Actions window.
- Internal technical documentation such as `ACCESSIBILITY.md` may remain English.
